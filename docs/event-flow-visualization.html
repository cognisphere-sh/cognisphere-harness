1 · The big picture

One Node process owns many agents. Each agent has its own SQLite queue + worker pool and spawns one pi child per batch.

Unit of work

A batch = all queued rows for one threadId, concatenated into a single prompt. Threads are serialized; different threads run concurrently up to maxConcurrentSlots.

Thread identity

threadIdStrategy: single→"default", plugin→<pid>, plugin_channel→<pid>:<chan>. One pi session JSONL per thread.

Durability

Every event is a row first, action second. A crash mid-batch leaves rows recoverable via sweepInFlight() at next boot.

2 · End-to-end flow of one event

From plugin signal to terminal status. File references are to apps/server/src/.

1
Plugin receives external signal
A webhook hit, poll result, or cron tick. The plugin calls ctx.notify("message_received", {text, channelId, metadata, …}). The context wrapper injects pluginId and metadata._notification = name, then calls runner.notify().
2
Runner enqueues — durable first runner.ts:286
notify() computes threadId (via strategy or threadIdOverride), then db.enqueue() inserts a row with queued, attempts=0. The row exists before any decision about how to deliver it.
3
Steer-or-wake decision runner.ts:301
Is there an active streaming batch on the same thread, with doNotSteer !== true?
• Yes → build <harness-metadata>+text, rpc.sendSteer() into the live child, push a single-row group onto steerGroups, markInFlight([id]).
• No → signalAll() wakes an idle worker. (Silent / doNotSteer rows never wake a worker — they ride the next batch.)
4
Worker claims a thread runner.ts:364
peekHighestPriorityThread(exclude=active) returns a thread with queued, non-silent rows no other worker owns. Then the continue-first claim: dequeueBatch(threadId, continueOnly=true). If it yields rows → continue batch; else dequeueBatch(threadId) claims all resend+fresh rows. Claimed rows flip to in_flight in one transaction.
5
Resolve session + spawn pi runner.ts:391, 649
getThreadSessionId() or generate a randomUUID() and persist in threads. spawnPi() builds argv: --session <dir>/<uuid>.jsonl --provider --model --thinking --tools read,bash,edit,write,grep,find,ls --system-prompt … --extension <agentDir>/extensions/harness-bridge.ts. Env = process.env ⨁ provider creds ⨁ flattened secrets ⨁ subagent model vars.
6
Wire the live event stream runner.ts:455
rpc.onUserMessageStart() → active.deliveredCount++ (counts deliveries in dispatch order). rpc.onHarnessEntry() → for each delivered row, db.setRowEntryId(id, sessionId, entryId) — writes pi_entry_id in real time, even before exit, even on rows that later fail.
7
Assemble prompt & send runner.ts:467
Continue rows → one buildContinuationNudge() (not re-rendered). Fresh/resend rows → each renders <harness-metadata>+text (resend carries Retry: true when attempts>0). rpc.sendPrompt(); phase → streaming. On a continue batch, drainQueuedAsSteers() injects the remaining resend+fresh rows as one combined steer.
8
Await completion or crash runner.ts:502
Promise.race([agentEnded, rpc.waitExit()]) — guards against pi dying before emitting agent_end. If active.cancelled after the race → throw into the cancel branch. Otherwise endStdin(), await exit (5s SIGKILL guard).
9
Verdict from message shape runner.ts:515, endOfTurn()
No JSONL read. endOfTurn(messages): complete iff last message is assistant with stopReason:"stop". Else tagged [agent_error] / [incomplete]. No agent_end at all → [crash] + stderr tail.
10
Delivery split & terminal status runner.ts:582, finalizeBatch()
Rows split by whether they reached the model (deliveredRowIds()):
• never delivered → markBatchFailed [not_delivered] (retry as resend).
• delivered + complete → done.
• delivered + incomplete → markBatchFailed with the tag (retry as continue, since pi_entry_id is already set).
Finally emit("batch-completed") for any pending stale-swap.

4 · Key decision points

Every branch that changes how an event is handled. These are the load-bearing forks in the code.

D1 · Steer or wake? notify() runner.ts:301
Active streaming batch on this thread AND !doNotSteer AND active.rpc?
→ Steer into the live child (fire-and-forget) + markInFlight.
→ Else wake a worker via signalAll().
D2 · Silent / doNotSteer suppression
peekHighestPriorityThread filters is_silent=0. A silent row never wakes a worker on its own — it waits for the next non-silent batch on the same thread. Used for scheduler late-firings.
D3 · Continue-first claim runner.ts:383
dequeueBatch(threadId, continueOnly=true) first. If it returns rows → continue batch (prompt = nudge only); resend+fresh rows are steered in afterward. Continue rows are never mixed into a concatenated prompt.
D4 · Retry mode per row deriveRetryMode() queue.ts:73
pi_entry_id != null → continue (delivered; nudge, don't resend).
attempts > 0 → resend (never delivered; full text + Retry:true).
else → fresh.
D5 · Completion verdict endOfTurn() runner.ts:123
Last message assistant + stopReason:"stop" → complete.
stopReason:"error" → [agent_error].
toolResult / toolUse / length → [incomplete].
no agent_end → [crash].
D6 · Delivery split deliveredRowIds() runner.ts:567
Per row: did its user-message index actually arrive (deliveredCount)? Undelivered rows fail as [not_delivered] (→ resend) even if the rest of the batch completed.
D7 · attempts vs maxAttempts markBatchFailed queue.ts:340
attempts+1 ≥ maxAttempts (default 3) → failed (DLQ). Otherwise bounce to queued.
D8 · Cancel vs fail processBatch catch runner.ts:529
In the catch branch, active.cancelled → markBatchCancelled (terminal, no retry). Else → markBatchFailed (with stderr tail + [runner_error] fallback tag).

5 · How retry works — the two modes

The central design insight: there is no retry_mode column. The mode is derived from pi_entry_id, which is written in real time during the batch — so it survives a full server crash.

Why it works: pi_entry_id is set the instant a row's user message reaches the model (via the harness-bridge extension → setRowEntryId), not at exit. So even a row whose batch later crashes carries the marker. At the next dequeue, its presence answers "was this already said to the model?" — and that single bit chooses continue vs resend.

🔁 CONTINUE mode

pi_entry_id != null — the row's message already reached the model; the turn was cut off after delivery.

  • Text is already in pi's session history — do NOT resend.
  • Prompt = a single buildContinuationNudge() tagged Continuation: true.
  • "The automated equivalent of a human typing continue."
  • Claimed in isolation via dequeueBatch(…, continueOnly=true).
// buildContinuationNudge — slim, no plugin/channel
<harness-metadata>
Timestamp: …
Continuation: true
</harness-metadata>
Your previous turn ended before it finished…
Continue from where you left off.

↩️ RESEND mode

pi_entry_id IS NULL & attempts > 0 — the row was requeued but never delivered to the model.

  • Full original text is re-rendered.
  • Carries Retry: true (because attempts > 0).
  • Base prompt tells the agent: "you may have done partial work — don't blindly redo it."
  • Concatenated into the normal batch prompt.
// buildHarnessMetadata when attempts > 0
<harness-metadata>
Timestamp: …
Plugin: telegram
Channel: 12345
Retry: true
</harness-metadata>
<full original message text>

Bug it prevents: if requeueFailed / setStatus(queued) cleared pi_entry_id, a delivered-then-dead-lettered row would resend and duplicate a message already in pi's history. The column is deliberately omitted from those SET clauses.

7 · How pi-coding-agent extensions work

The harness controls exactly what pi loads. --no-extensions disables auto-discovery; explicit --extension <path> still loads. The one seeded extension — harness-bridge — closes the real-time entry-id loop.

Loading model — spawnPi() runner.ts:649

  • --no-extensions --no-skills --no-prompt-templates --no-themes --no-context-files suppress pi's default discovery — the harness controls the surface.
  • Skills: one --skill <skills-root> — pi's loadSkillsFromDir recurses for SKILL.md.
  • Extensions: pi's loadExtension does not recurse — it expects index.ts/.js / package.json at the path. So the runner passes one --extension per first-level child of extensions/, and verifies the entrypoint exists first (a missing module crashes the whole pi run).
  • harness-bridge is not special-cased — it ships as an ordinary seeded agent extension at extensions/harness-bridge.ts, copied at agent-create from templates/base/extensions/.

harness-bridge — the entry-id reporter harness-bridge.ts

Its sole job: report each user-message session entry id back to the parent in real time, so the harness can link a queued row to its place in pi's JSONL without reading the file and without waiting for exit (a failed batch never reaches a post-exit read).

Transport

  • ctx.ui.setStatus("pi-harness", <json>)
  • In RPC mode → a fire-and-forget extension_ui_request{method:"setStatus"} frame on stdout.
  • PiRpcClient.handleExtensionUi intercepts statusKey==="pi-harness", parses {kind,index,entryId}, routes to onHarnessEntry.
  • Other dialog methods (select/confirm/input/editor) are auto-cancelled — no human to answer.

Why a sweep, not a read?

  • pi emits the extension event before it appends the message to the session (emit, then appendMessage) — so the entry id doesn't exist at message_end yet.
  • Instead the extension re-scans getEntries() on every event (agent_start, message_start, turn_start/end, agent_end) and emits newly-appeared user entries by index.
  • The extension knows nothing about harness row ids — it reports {index, entryId}; the parent maps index→row.
export default function harnessBridge(pi) {
  let reported = 0;                              // per-run counter
  const sweep = (ctx) => {
    const userEntries = ctx.sessionManager.getEntries()
                          .filter(isUserMessageEntry);
    for (let i = reported; i < userEntries.length; i++)
      ctx.ui.setStatus("pi-harness",
        JSON.stringify({ kind:"user_entry", index:i, entryId:userEntries[i].id }));
    reported = userEntries.length;
  };
  pi.on("agent_start", (_e,ctx)=>{ reported=0; sweep(ctx); });
  pi.on("message_start", (_e,ctx)=>sweep(ctx));   // + turn_start/end, agent_end
}

10 · Critical code blocks

The handful of blocks where the whole design lives.

Continue-first dequeue + delivery split runner.ts:workerLoop / finalizeBatch

// 1. Continue rows resume in isolation — their own nudge prompt
const continueBatch = db.dequeueBatch(threadId, true);   // continueOnly
const isContinue   = continueBatch.length > 0;
const batch        = isContinue ? continueBatch : db.dequeueBatch(threadId);

// 2. After agent_end: who actually reached the model?
deliveredRowIds(active) {
  const delivered = new Set(active.continueIds);          // prior-batch rows
  if (active.deliveredCount >= 1) // prompt landed (user index 0)
    active.promptRowIds.forEach(id => delivered.add(id));
  for (let k = 1; k < active.deliveredCount; k++)        // each steer
    (active.steerGroups[k-1] ?? []).forEach(id => delivered.add(id));
  return delivered;
}

// 3. Split: undelivered → resend; delivered → done | continue
if (undelivered.length)  db.markBatchFailed(undelivered, "[not_delivered] …", max, sid);
else if (complete)        db.markBatchDone(deliveredIds, sid);
else                     db.markBatchFailed(deliveredIds, errorTag, max, sid);

Verdict from message shape — no JSONL read runner.ts:endOfTurn

function endOfTurn(messages) {
  const last = messages[messages.length - 1];
  if (last?.role === "assistant" && last.stopReason === "stop")
    return { complete: true,  errorTag: "" };               // ✓ natural stop
  if (last?.role === "assistant" && last.stopReason === "error")
    return { complete: false, errorTag: `[agent_error] …` };  // model/API error
  return { complete: false, errorTag: `[incomplete] … (last=…)` }; // toolResult/length
}

Retry-mode derivation — the one bit that decides everything queue.ts:deriveRetryMode

function deriveRetryMode(row) {
  if (row.pi_entry_id != null) return "continue";  // delivered → nudge
  if (row.attempts > 0)        return "resend";    // requeued, never delivered
  return undefined;                              // fresh
}
