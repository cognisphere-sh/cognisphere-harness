hermes-agent vs CogniSphere

Two agent harnesses, two philosophies. One is a mature, batteries-included production platform with ~989K lines of Python. The other is a lean ~15K-line TypeScript orchestration layer built on fundamentals. This report grades both against the seven things you actually care about.

Dimension by Dimension

The seven criteria, in detail

For each criterion: how each system actually implements it, where in the source it lives, and who wins.

1

Minimal, simple, easy to modify & extend

How much you must understand before you can safely change things

hermes-agent

A mature platform with breadth as its design goal. The agent loop alone lives in agent/conversation_loop.py (~264KB) and run_agent.py, with the gateway at gateway/run.py (~24K LOC). Extension is well-abstracted — register a tool, subclass gateway/platforms/base.py — but the surface is vast: 109 modules in agent/, 126 plugins, 99 tool files.

Extensible? Absolutely. Minimal? No. Onboarding means reading dozens of files plus the 1,158-line AGENTS.md rulebook.

CogniSphere

Built on fundamentals: filesystem, SQLite, subprocess, HTTP. The entire server is ~20 files / 3.8K LOC — main.ts, runner.ts, queue.ts, agent-manager.ts, plugin-registry.ts. No orchestration framework, no DI container, no magic.

You can read the whole core in an afternoon and confidently change it. The deliberate trade: it offloads the LLM loop to the external pi binary, so the harness stays an orchestration shell.

VerdictPi wins decisively (9 vs 4). This is your #1 priority and it's the one property that can't be bolted on later.
2

Adding new event / notification sources

Wiring in chat, email, cron, webhooks as input channels

hermes-agent

34 platform adapters already shipped. A dynamic gateway/platform_registry.py auto-registers anything subclassing the adapter ABC. Beyond messaging there's a durable cron daemon (cron/scheduler.py), webhooks with HMAC auth, an API server, and Hermes-as-MCP-server (mcp_serve.py). Plugin lifecycle hooks: on_session_start, pre_llm_call, etc.

CogniSphere

One clean abstraction: implement the Plugin interface (types.ts) and call ctx.notify(event, payload). Drop a folder under plugins/<id>/ and plugin-registry.ts dynamic-imports it. Four ship today (admin, scheduler, telegram, gws/Gmail). Adding one is a single TS file — but you're building sources, not picking from 34.

VerdictHermes wins on batteries-included reach (9 vs 8); pi wins on per-source authoring simplicity. If you need Slack today, hermes has it; if you need a custom source, pi is faster to write.
3

Session management by source & thread

Keying conversations to platform, channel, thread id

hermes-agent

Sessions keyed {user_id}@{platform}#{thread_id}, persisted to a versioned SQLite store (hermes_state.py, schema v14) with FTS5 full-text search. In-memory LRU cache of 128 live agents with a 1-hour idle TTL. Compression creates parent_session_id lineages so split sessions stay linked. Forum topics map to separate sessions automatically.

CogniSphere

Routing is explicit and configurable per agent via threadIdStrategy: single, plugin, or plugin_channel (${pluginId}:${channelId}). A unified SQLite events table tracks every notification with thread_id, status, priority, plus a per-thread worker lock. Fully durable across restarts (no in-memory session store to lose).

VerdictPi edges it (9 vs 8) — the declarative threadIdStrategy is exactly the source/thread keying you asked for, with less code. Hermes is more battle-tested and adds FTS search.
4

Self-evolving

Agent writes, curates, and prunes its own skills/tools over time

hermes-agent

The standout feature. A Curator (agent/curator.py, ~1.8K LOC) runs in the background tracking per-skill usage in .usage.json and auto-transitions agent-created skills active → stale → archived (never deletes; restorable, pinnable). The agent writes/patches its own skills mid-run via a skill_manage tool. Trajectory capture (trajectory_compressor.py) preps data for fine-tuning the next model generation.

CogniSphere

The framework is designed for it — agents can author skills/agent/, scripts/agent/, and extensions/agent/ on disk — but the autonomous loop (admin agent + platform plugin that CRUDs skills/agents) is explicitly deferred in v0 (docs/v0-deferred.md). No usage-driven curation, no auto-archival today. Static markdown skills, hand- or agent-written, discovered at runtime.

VerdictHermes wins big (8 vs 3). Its curator is a genuinely differentiated, working self-maintenance loop. Pi has the bones but not the muscle yet.
5

Memory systems & context compaction

Summarization, durable recall, token-budget management

hermes-agent

The widest gap in the report. agent/context_compressor.py (~2K LOC) does token-budget compaction: protects head + tail, summarizes the middle with a cheap auxiliary model, prunes verbose tool output first, merges iteratively, and stays between cache boundaries so prompt caching survives. Memory is pluggable across 8 backends (honcho default, mem0, supermemory…) with per-turn sync_turn + prefetch. Cross-session recall via FTS5 + LLM summarization.

CogniSphere

Manual and file-based. Each agent owns a workspace/ (and workspace/memory/) it edits with read/write tools — persistent and shared across threads, but the agent manages the schema itself. No automatic compaction: no token counting, no summarization, no sliding window. The scaling answer is "delegate to a sub-agent for long reads." Compaction, if pi does it at all, happens inside the external pi binary, not the harness.

VerdictHermes wins by the widest margin (9 vs 3). Its compressor is production-quality and cache-aware — exactly what you want, and the single best thing to port to pi.
6

Persistent sub-agents

Durable delegation that survives across batches

hermes-agent

Delegation (tools/delegate_tool.py, ~2.8K LOC) is rich — parallel batch tasks, leaf vs orchestrator roles, depth/concurrency caps — but synchronous: the parent blocks and children die if it crashes. True persistence comes from the separate Kanban board (plugins/kanban/): a SQLite-backed dispatcher daemon with stale-claim recovery and failure-blocking. Two different mechanisms for two needs.

CogniSphere

Sub-agents are pi -p processes spawned via a scripts/agent/subagent wrapper. Persistence is elegant by convention: reuse the same --session-dir with -c and the child resumes its prior JSONL across many parent batches. Sessions nest under sessions/<thread>/subagents/<id>/. Lightweight, not first-class — no shared queue or peer messaging, communication is stdout only.

VerdictPi edges it (7 vs 6) for your phrasing specifically — "persistent" sub-agents resumable by session-dir is built into its model. Hermes is more powerful but its delegation is synchronous; persistence needs the heavier Kanban subsystem.
7

Ease of deploying multiple agents

Standing up and operating a fleet

hermes-agent

Hardened single-container deploy: Dockerfile + s6-overlay supervision, multi-arch, UID/GID remapping, docker-compose.yml (gateway + dashboard). But the unit of deployment is a profile per container — many agents means many HERMES_HOME volumes / containers. No built-in federation; each gateway is independent. Production-solid, but heavier per agent.

CogniSphere

One Node process hosts many agents in-memory, each isolated on disk under agents/<id>/. Adding an agent = drop an agent.json + restart (or POST /api/agents/:id/start). Back up an agent by tar-ing its folder. Horizontal scale = run more servers with different COGNISPHERE_ROOT_DIR. Dead simple — though all agents share one process (no per-agent isolation), and containerization is a v1 item.

VerdictPi wins clearly (9 vs 6). Multi-agent is its native shape; hermes treats each agent as a heavier, more isolated unit.

Recommendation

Build on CogniSphere — and steal from hermes

You ranked "minimal, simple, easy to modify and extend" first, and you want multi-agent deployment, source/thread sessions, and persistent sub-agents. Pi already wins four of those — and they're the structural properties you can't retrofit. The two real gaps (memory/compaction, self-evolution) are additive features that hermes has already designed and proven, so you can port the ideas onto a clean base.

Why pi is the right foundation for your goals

What pi gives you out of the box

  1. Criterion 1 (simplicity): 15.5K legible LOC vs 989K — the bet that's hardest to undo.
  2. Criterion 2 (sources): a one-file Plugin + ctx.notify contract.
  3. Criterion 3 (sessions): declarative threadIdStrategy keyed by source/thread.
  4. Criterion 6 (sub-agents): session-dir-resumable child pi processes.
  5. Criterion 7 (fleet): many agents per process, add one with an agent.json.

The 2 gaps — port these from hermes

→
Context compaction (criterion 5). Port the strategy in agent/context_compressor.py: protect head+tail, summarize the middle with a cheap model, prune tool output first, keep it cache-safe. Highest ROI item.
→
A memory layer (criterion 5). Pi's workspace/memory/ is the storage; add hermes-style per-turn sync_turn + prefetch over it (even just embeddings + the SQLite you already have).
→
A curator loop (criterion 4). Port the .usage.json + active→stale→archived lifecycle from agent/curator.py onto pi's skills/agent/ dir. Pi already has the files; it needs the maintenance loop.
The decision rule: structural properties (simplicity, multi-agent, clean plugin/session models) are nearly impossible to add later — pi already has them. Intelligence features (compaction, memory, curation) are self-contained modules — hermes proves the design, you port ~3 of them. So: start from CogniSphere, and treat hermes-agent as a reference implementation to mine. The exception is if you need a polished omni-channel (Telegram/Slack/WhatsApp) assistant shipping this week — then run hermes and accept the weight.
