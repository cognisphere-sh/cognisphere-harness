Executive summary

The harness is a well-conceived multi-agent platform: a small, genuinely clean plugin contract (manifest + start(ctx) + stop() with ctx.notify() as the single outbound primitive), a durable per-agent SQLite queue, and a per-batch pi --mode rpc child model. The bones are good — the lost-wakeup-free worker wait loop, the U+2028-safe RPC framing, and the live entry-id capture design are all things many teams get wrong and this codebase gets right.

The risk is concentrated in four places that can silently lose or misroute a user's message, plus a same-origin XSS that turns the (intended) web UI into a full account-takeover vector. None of these are visible in normal happy-path testing, which is exactly why they're dangerous. Below them sits a broad layer of medium-severity reliability gaps (no batch watchdog, no missed-fire recovery, swallowed startup errors) and a clear, high-leverage structural opportunity: ~70% of the bugs in the four plugins live in boilerplate the plugin contract doesn't provide — a thin Plugin SDK would erase most of them at once.

Architecture at a glance

One Node process owns everything: the plugin registry, every agent's SQLite queue, all in-process plugin instances, and the HTTP server. Each agent spawns one short-lived pi child per batch.

┌─ Node process ─────────────────────────────────────────────┐
  HTTP (Hono: /api /admin /healthz · raw dispatch: /webhook/…)
        │
        ▼
  AgentManager ── SecretsStore · ModelsStore · PluginRegistry
    └─ agents: Map<id, AgentInstance>
         ├─ agentJson, state, db: AgentDb  ← outlives the runner
         ├─ plugins: Map<id, PluginEntry>   in-process actors
         └─ runner: AgentRunner          ← ephemeral, rebuilt on reload
              ├─ workers (Promise[])      per-thread exclusion
              └─ active: Map<threadId, ActiveBatch>
                        │
                        ▼ spawn per batch
                 pi --mode rpc  (JSON-RPC over stdio, dies on agent_end)

The single most consequential structural observation in this audit: the durable queue (AgentDb) deliberately outlives the runner, but the only write path into it goes through the runner's notify(), which throws while the runner isn't running. That inversion is the root cause of two separate message-loss windows (boot and reload). See R-2 and the action plan.

① Critical & data-loss findings

These are silent — they don't throw, don't log loudly, and don't show up in single-message testing. They corrupt state or lose work on the second message, on a retry, on a restart, or on an error path.

Critical ✓ Resolved C-1 · harness-bridge reports session-absolute entry indexes; the runner maps them batch-relative

apps/server/agents/templates/base/extensions/harness-bridge.ts:30,37,45,51 · runner.ts:457,563 · queue.ts:285

The bridge resets reported = 0 on every agent_start and then sweeps all user entries in the session file, emitting absolute index i:

pi.on("agent_start", (_e, ctx) => { reported = 0; sweep(ctx); });
// sweep: for (let i = reported; i < userEntries.length; i++)
    ctx.ui.setStatus(KEY, JSON.stringify({ kind:"user_entry", index: i, entryId: entry.id }));

But each thread reuses one session JSONL across all its batches (--session <sessionId>.jsonl), and pi's getEntries() returns the entire loaded file. So on batch ≥2 of any thread — i.e. the second message of every conversation — the sweep re-reports the historical user entries at indexes 0..n-1, and the runner writes a batch-1 entry id onto batch-2's rows (setRowEntryId updates unconditionally). The real new prompt arrives at index n, maps to no group, and its true entry id is dropped.

Because smart-retry classifies a row as "already delivered → continue (generic nudge, original text never re-sent)" purely from pi_entry_id IS NOT NULL, a row that never reached the model can be pre-marked delivered and, on failure, silently dropped instead of resent. This is data loss masquerading as a successful retry.

Fix: capture a baseline user-entry count at extension load (before any prompt) and report index − baseline, never emitting entries that predate the run — or have the runner record the pre-spawn user-entry count and subtract it. Add a guard in setRowEntryId rejecting an index already bound.
✓ Resolved. Chose the ordering-independent strategy: the runner now ignores the bridge's absolute index and binds each genuinely-new entryId to the next message group in arrival order, deduped via ActiveBatch.seenEntryIds (pre-seeded from the new queue.entryIdsForThread()). setRowEntryId gained an AND pi_entry_id IS NULL guard so a binding is never overwritten. Verified against vendored pi that the prompt is appended at message_end (after agent_start), confirming the approach holds regardless of event timing. Unblocks the §5 smart-retry work.
runner.ts (onHarnessEntry, ActiveBatch) · queue.ts (entryIdsForThread, setRowEntryId guard) · bridge unchanged · npm run check ✓

Critical ✓ Resolved C-2 · Runner error paths never kill/await the pi child → leaked processes + same-session double-spawn

apps/server/src/runner.ts:514 (happy-path guard) vs 530–553 (catch — no kill)

The happy path escalates to SIGKILL after 5s (const exitTimer = setTimeout(() => rpc.kill("SIGKILL"), 5000)). The catch path does neither — on [runner_error] (e.g. sendPrompt rejected while the child is alive and healthy) or on abort, it marks the batch failed/cancelled and falls through to active.delete(threadId) in the surrounding flow without ending stdin or awaiting exit:

} catch (err) {
    // marks batch failed/cancelled… and that's it.
    // child is never endStdin()'d, never killed, never awaited.
}

Two consequences: (a) a zombie pi process is abandoned; (b) freeing the thread lets a worker immediately dequeue the same thread and spawn a second pi against the same --session JSONL while the first is still appending — concurrent write corruption of the conversation file.

Fix: wrap processBatch in a finally that, if the child hasn't exited, calls endStdin() then await Promise.race([rpc.waitExit(), timeout]) with SIGKILL escalation before the thread is released from active.
✓ Resolved. Added an ensureChildExited helper (end stdin, await exit, 5s SIGKILL) and a finally in processBatch that runs it on every exit path — success, runner_error, abort, throw — before the worker loop releases the thread from active. Idempotent via a childExited flag (no double-kill/await); relies on rpc.ts's cached exit promise and try/catch-wrapped kill/endStdin so teardown is safe on an already-dead child. Closes both the zombie leak and the concurrent-JSONL-write window.
runner.ts (ensureChildExited, childExited flag, processBatch finally) · npm run check ✓

Critical ✓ Resolved C-3 · peekHighestPriorityThread priority ordering is broken (verified inversion)

apps/server/src/queue.ts:198–210
SELECT DISTINCT thread_id FROM events
WHERE status='queued' AND is_silent=0
ORDER BY priority DESC, id ASC

SQLite permits ORDER BY on a non-selected column under DISTINCT and uses an arbitrary row's priority per thread. Empirically: thread A with rows (prio 0, prio 9) vs thread B (prio 5) returns B first — the thread holding the globally highest-priority message loses. Latent today (no plugin sets a non-zero priority yet) but the column and the documented contract both exist, so the first plugin that uses priority will hit a silent scheduling bug.

Fix: SELECT thread_id, MAX(priority) p, MIN(id) first_id FROM events WHERE status='queued' AND is_silent=0 GROUP BY thread_id ORDER BY p DESC, first_id ASC.
✓ Resolved. Applied the GROUP BY query exactly as suggested. Verified with a standalone better-sqlite3 repro: the A=(prio 0, prio 9) vs B=(prio 5) case now returns A (was B). dequeueBatch's within-thread ordering was already correct, so no change there.
queue.ts (peekHighestPriorityThread) · repro confirmed · npm run check ✓

Critical C-4 · Stored XSS → full account takeover via raw file serving

apps/server/src/api/files.ts:115–135, 223–228

GET /:id/fs/raw serves agent-dir files inline with a sniffed mime type — image/svg+xml for .svg, text/html for .html:

const disposition = c.req.query("download")
    ? `attachment; filename="${filename}"`
    : `inline; filename="${filename}"`;  // default

Agent-dir files are written by an LLM that ingests untrusted external input (email bodies, Telegram messages). A prompt-injected instruction that makes the agent write payload.html into its workspace yields script executing on the harness origin when the operator opens the link — with the pi_sid cookie attached to same-origin fetches, i.e. full authenticated control of every /api/* route (read secret structure, rewrite agent.json, delete threads). The filename is also interpolated unescaped into content-disposition.

Fix: force attachment disposition (or add Content-Security-Policy: sandbox + X-Content-Type-Options: nosniff) for text/html and image/svg+xml; quote/strip " in the filename.

② Security findings

High S-1 · Corrupt secrets.json read as {} → next save permanently wipes all secrets

apps/server/src/api/secrets.ts:177 (catch→{}) + 137 (non-atomic write)

readSecrets swallows parse errors (catch { return {}; }). PUT /api/secrets reads existing, merges the request body, then writeFileSyncs — a plain write, not write-temp-then-rename. So a crash mid-write truncates the file; the next save then reads it as empty and rewrites it containing only the keys in that request, silently destroying every other agent's credentials. Note the inconsistency: the runtime SecretsStore.load() and ModelsStore.load() both throw on bad JSON — only the secrets API treats corruption as "empty".

Fix: make readSecrets throw (→500) on unparseable non-empty files, and write atomically (writeFileSync(tmp) + renameSync) for secrets.json, models.json, users.json, agent.json, and plugin configs. Route the secrets API through SecretsStore rather than re-reading the file.

High S-2 · Telegram document file_name used unsanitized → path traversal write primitive

apps/server/plugins/telegram/index.ts:259–261, 285–287

file_name is set by whoever sends the document and flows straight into join(inboxDir, filename). With the default empty allowedChatIds, anyone can send ../../system_prompts/0-base_prompt.md and overwrite files in the agent dir — a prompt-injection-grade write primitive. Separately, all attachments share one flat inboxDir keyed by message_id (unique only per chat), so files from two chats overwrite each other. (gws avoids this with per-message subdirs.)

Fix: basename(file_name) + the same sanitizeFilename gws uses, and write under inbox/<chatId>/<messageId>/.

High S-3 · Symlink escape defeats resolveSafe containment on every /fs/* route

apps/server/src/api/files.ts:195–201

The containment check is purely lexical (normalize + relative + startsWith("..")), with no realpathSync. A symlink inside the agent dir (creatable by the agent's own shell tools) pointing outside lets the authenticated UI read, overwrite (PUT), and recursively delete (DELETE …?recursive) targets outside the dir — .secrets/secrets.json, pi's plaintext ~/.pi/agent/auth.json OAuth tokens. docs/api.md:476 claims containment the code doesn't deliver.

Fix: after the lexical check, realpathSync the deepest existing ancestor and re-verify it's under realpathSync(root); or lstat and refuse to follow symlinks.

Medium S-4 · No login throttling; plaintext passwords; default admin/changeme never enforced away

apps/server/src/api/auth.ts:37–39, 106–114, 188–202

POST /api/auth/login is public with zero rate-limiting or lockout, passwords are stored plaintext, and the file auto-creates as admin/changeme. A server the project's own warning calls "exposed" is brute-forceable at network speed against a known default. Plaintext-at-rest is a documented v0 tradeoff; the missing throttle and the absence of a startup refusal (or loud /healthz flag) while the placeholder credential is live are not.

Fix: in-memory per-IP/username failure counter with exponential backoff; hash passwords (even unsalted scrypt is a one-file change); refuse to bind non-loopback while admin/changeme still verifies.

Medium S-5 · Webhook dispatcher leaks distinct 404 bodies → unauthenticated agent/plugin enumeration

apps/server/src/api/webhook.ts:37–51 · auth.ts:219–226 (cookie missing Secure)

Distinct unauthenticated 404s (unknown agent: <id> vs unknown plugin or no http handler: <pid>) let anyone enumerate valid agent ids and installed plugins by prefix scanning, with no body-size cap or rate limit before the plugin sees the request. Separately, the session cookie sets SameSite=Lax but not Secure, so under TLS termination the cookie still travels over a plain-HTTP same-host request; CSRF rests entirely on Lax.

Fix: single constant 404 body for all webhook miss cases; set secure on the cookie when the request is HTTPS; optional per-plugin shared-secret path segment.

③ Reliability & lifecycle findings

High R-1 · No batch watchdog — a hung pi child holds a worker slot forever

apps/server/src/runner.ts:493 (await rpc.sendPrompt), 505 · rpc.ts:177

sendPrompt and the Promise.race([agentEnded, waitExit()]) have no timeout; pending promises reject only on child close/error. A child that is alive but unresponsive (wedged model call, deadlocked extension) blocks processBatch indefinitely — and with the default maxConcurrentSlots: 1, that stalls the entire agent until an operator manually aborts.

Fix: a configurable per-batch deadline (agentJson.batchTimeoutMs) that triggers the abort→SIGKILL path. The HLD already specifies "after 30 min hard-kill" — it just isn't implemented.

High R-2 · Notification loss windows — ctx.notify silently drops messages when the runner is null or not yet started

apps/server/src/agent-manager.ts:605–616 · runner.ts:284

Two windows: (a) boot — plugins are start()ed before runner.start(), so anything a plugin delivers during its own startup (a long-poll that already consumed messages from the external service) hits runner.notify's throw "not running", is logged, and lost. (b) reload/stale-swap — stopAgent sets inst.runner = null; the optional chain inst.runner?.notify(...) makes it a silent no-op, not even an error. The queue is alive and persistent through both windows; only the write path is gone.

Fix (root cause): move thread-id computation + durable enqueue out of the ephemeral runner up to AgentInstance/manager level (the runner then only consumes the queue and handles live steering). This makes notifications loss-free by construction. See P-2.

High R-3 · Scheduler has no missed-fire recovery — reminders during downtime silently evaporate

apps/server/plugins/scheduler/index.ts:96–110

reload() just constructs new Cron(...) per schedule. Croner only schedules future occurrences; nothing persists last-fire/next-fire and start() does no catch-up. A one-shot reminder whose moment passes during a restart never fires — not even later, because nobody records that it was missed. For a reminder system this is a core correctness gap. Compounding it: fire() has no try/catch and croner isn't set catch:true, so a notify() throw (runner swapping) escapes into croner's timer context and the fire is lost.

Fix: persist lastFiredAt per schedule; on start, compare Cron.nextRun(lastFiredAt) to now and fire missed occurrences (at minimum for onetime schedules). Wrap fire() in try/catch.

High R-4 · Telegram: in-memory offset → duplicate delivery after every restart; timeoutMs ignored → silent poll stall

apps/server/plugins/telegram/index.ts:134, 153, 191–208, 305

Three compounding issues. Offset is private offset = 0, never persisted; the startup comment claims it "drops pending updates" but the code only calls getMe. After a restart with recent traffic, every update Telegram still holds (up to 24h) is redelivered. Timeout — the poll loop passes timeoutMs but api() never reads it, so a wedged getUpdates connection stalls the loop forever with no log. Loss — offset is advanced before handling, so a transient notify/enqueue failure is swallowed and the message is gone.

Fix: persist offset to state/offset.json after each batch; honor timeoutMs via AbortSignal.timeout; distinguish transient enqueue failure (don't advance offset) from a malformed-update poison pill.

High R-5 · gws default mode drops non-latest unread mail, then marks it all read (unrecoverable)

apps/server/plugins/gws/index.ts:344–351, 434–437

Default mode only ever considers the thread's latest message (const last = ordered[ordered.length-1]) but marks every unread message in the thread read. Two arrivals between polls where the latest is Cc-only: the earlier message addressed to the agent is never delivered and is permanently marked read. There's also no dedupe ledger in default mode, so a crash between notify and markRead re-delivers.

Fix: emit every unread message addressed to the agent (wake on the newest, older ones silent); only mark read what was actually emitted; consult a MessageId-keyed seen-set in default mode too.

Medium R-6 · agent.json is never schema-validated → malformed config fails deep in the batch path

apps/server/src/agent-manager.ts:366 (bare cast) · runner.ts:681, 801

Plugin configs get ajv validation; AgentJson itself is a bare JSON.parse(raw) as AgentJson. A missing model survives start and then throws Cannot read properties of undefined in spawnPi on every batch, burning retries to the dead-letter queue. A typo'd threadIdStrategy falls through computeThreadId's switch with no default, failing the SQLite bind inside notify → caught by R-2's wrapper → message lost.

Fix: ajv-validate AgentJson in step 1 of startAgent (ajv is already constructed); add a default: throw to computeThreadId.

Medium R-7 · Several startup paths throw outside the catch → agent left "stopped" with error: null

apps/server/src/agent-manager.ts:424–433 · telegram/index.ts:123–145 (opposite failure mode)

In startAgent, the mkdirSync + AgentDb constructor + backfillThreadSessions sit outside the catch-and-record pattern that steps 1–2 use carefully. If the DB file is corrupt/locked they throw, startAgent rejects, loadAgent only logs, and the UI shows a healthy "stopped" agent with no hint why. Telegram has the opposite failure mode: missing/bad token → warn + return, so start() resolves and the plugin is recorded running while delivering nothing. (gws does it right — it throws detailed errors.) The codebase needs one rule: fail-to-start = throw.

Fix: wrap the remaining startAgent steps in the catch-and-record pattern; make telegram throw on getMe failure and declare required: ["TELEGRAM_BOT_TOKEN"].

Medium R-8 · Web: session expiry triggers an infinite login redirect bounce

apps/web/src/lib/api.ts:41 · App.tsx:66 · pages/login.tsx:24

The 401 handler navigates to /login but never flips the zustand auth store. LoginPage sees status === "authed" (stale) and <Navigate to="/">s back; the polled queries (3–5s) 401 again → /login → bounce indefinitely. The handler also fires for the login POST's own 401.

Fix: in the handler, useAuth.setState({ user:null, status:"anon" }) before navigating, and skip the handler for the login endpoint.
Also tracked (Low): re-entrant shutdown on a second SIGTERM (double db.close() throws) · failed stale-swap permanently wedges the runner in dequeuePaused with staleReason already cleared · fire-and-forget steers misbind rows if pi drops one · duplicate scheduler names leak ghost timers · scheduler state file has two uncoordinated read-modify-write writers (lost-update race).

Revamp: a Plugin SDK (highest structural leverage)

The core finding from the plugin audit: the contract is a B+ (small, clean, directory-as-identity), but the SDK is missing. About 70% of every plugin's bugs in this audit live in boilerplate the contract doesn't provide — each plugin re-implements the poll loop, state persistence, attachment handling, and secrets-missing behavior by hand, and they drifted into different wrong answers (telegram lost offset, gws lost non-latest mail, scheduler lost missed fires).

Adding a 5th plugin today means copy-pasting an existing one and inheriting its footguns. A thin SDK collapses that:

abstract class PollingPlugin<TConfig> implements Plugin {
  // owns: interval scheduling, stop flag, error backoff, structured logging
  abstract poll(ctx): Promise<void>;
}
ctx.state.readJson(name)            // typed, returns null if absent
ctx.state.writeJsonAtomic(name, v)   // tmp + rename, no torn writes
ctx.state.appendLedger(name, id)     // dedupe seen-set, cached
ctx.saveAttachment(name, buf)      // owns sanitize + per-source namespacing
                                  // + the <name>[relpath] inline-reference format

This single module would have prevented S-2 (telegram traversal), R-4 (offset loss), R-5 (gws mark-read), R-3 (scheduler missed fires), the two-writer state race, and the triple-duplicated formatTs — and would make a new plugin a manifest + a poll() body. Pair it with a documented "fail-to-start = throw" rule and a one-command seed installer (so the manual cp workflow in v0-deferred §3.1 stops being a footgun).

Roadmap alignment (docs/improvement-design.md)

The team's own forward design is sound and the audit reinforces its sequencing. Two notes where the audit changes the picture:

  • §5 Smart retries depends on C-1. The whole continuation-vs-replay mechanism keys off pi_entry_id IS NOT NULL — but the harness-bridge index bug means that flag is set on the wrong rows from the second message onward. Fix C-1 before building further retry logic on top of the entry-id signal, or the smart-retry feature will silently drop messages it believes were delivered.
  • §4 by-reference plugin layout (point pi at plugin seed/ dirs instead of copying into every agent) is the right call and it eliminates a whole class of install-time drift. It also pairs naturally with the SDK above — both move plugins from "copied artifacts" to "referenced source."
  • §1 memory & §2 compaction as pi extensions are clean, self-contained, and reversible — exactly the kind of work to do after the critical fixes land, since they don't touch the hot path.
  • §3 dreamer meta-agent is the big bet; the audit's note is just that it should consume the same usage/event data the retry-hygiene work (§5.3/5.4) makes legible — sequence it last, dry-run first, as the doc already says.
